설계 개요와 원칙

의료정보 시스템의 요구사항이 데이터 모델에 어떻게 반영됐는지 살펴봅니다.

설계 개요

네 가지 설계 원칙

Supabase Auth 중심 — 인증은 auth.users에 위임하고, 도메인 정보는 1:1 확장 테이블로 분리합니다.

역할(admin · doctor · patient)은 user_profiles에서 관리하고, 의사·환자의 고유 속성은 별도 테이블로 정규화합니다.

진료기록은 삭제 불가 — 오기입 시 is_corrected 표시 후 새 기록을 작성해 이력을 보존합니다.

모든 열람 행위는 access_logs에 Append-only로 기록해 의료정보 감사 추적을 보장합니다.

관계 구조

두 개의 도메인으로 분리

계정 도메인

누가 시스템을 쓰는가

users를 뿌리로 user_profiles(공통), doctors·patients(역할별)가 1:1로 확장됩니다. 의사는 departments를 참조해 소속 진료과를 갖습니다.

users ─ user_profiles
users ─ doctors ─ departments
users ─ patients

진료 도메인

무엇이 기록되는가

medical_records가 doctors와 patients를 잇는 중심 팩트 테이블이고, access_logs가 모든 열람 행위를 뒤에서 감사합니다.

doctors ─< medical_records >─ patients
medical_records ─< access_logs
